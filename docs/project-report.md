# Xiaomi Code 项目汇报

> AI 编程 Agent 技术预研与原型验证

## 一、项目背景

### 为什么已有 Claude Code 还要自己做？

这不是"重复造轮子"，而是**技术能力建设**。类比：

| 场景 | 用别人的 | 自己做的价值 |
|------|---------|-------------|
| 数据库 | 用 MySQL 就行 | 写一个迷你数据库，团队才真正理解索引、事务、WAL |
| 框架 | 用 React 就行 | 写一个迷你框架，才知道虚拟 DOM、Fiber、Hooks 的设计取舍 |
| AI Agent | 用 Claude Code 就行 | **写一个迷你 Agent，才掌握 Agent Loop、工具编排、安全防御的核心架构** |

**核心价值不是产品本身，而是过程中沉淀的能力：**

1. **深度理解 AI Agent 架构** — 不是调 API，而是理解 Agent Loop 为什么用双层 Generator、压缩为什么要 4 级管线、权限为什么要 deny-first
2. **掌握 AI 工程化关键技术** — Prompt Cache 优化（节省 90% token 成本）、流式并行工具执行、上下文窗口管理、错误恢复策略
3. **建立自主可控的技术储备** — 不依赖单一供应商，支持 5 个模型提供商，可接入任意 MCP 兼容工具
4. **形成可复用的技术资产** — 742 个测试用例、27 篇开发文档、完整的架构设计文档，可直接用于团队培训和技术分享

### 行业趋势

- GitHub Copilot 月活 1500 万+，AI 编程助手已成为开发者标配
- Claude Code 开源后，Agent 架构成为行业热点（每周 3400 万+ Explore 调用）
- MCP 协议成为 AI 工具生态标准，主流 IDE 和云服务商均已支持
- 掌握 Agent 架构 = 掌握下一代开发工具的核心技术

### 2.3 架构亮点

**1. 双层 Generator 架构**

```
用户输入 → QueryEngine（会话管理）→ query()（循环执行）→ StreamEvent
                ↑                        ↓
           消息持久化              工具执行 + 错误恢复
```

为什么不用简单的 while 循环？因为双层分离让会话管理（token 追踪、预算检查、压缩触发）和查询执行（工具调度、错误恢复、流式输出）各自独立演进，互不干扰。

**2. 4 级渐进压缩**

| 级别 | 策略 | 成本 | 效果 |
|------|------|------|------|
| Snip | 截断旧工具结果 | 零 API 调用 | 释放 30-50% |
| Micro | 压缩更旧的结果 | 零 API 调用 | 再释放 10-20% |
| Collapse | 折叠不活跃段 | 零 API 调用 | 再释放 10-15% |
| Auto | 模型生成摘要 | 1 次 API 调用 | 释放 60-80% |

前 3 级零成本，只有万不得已才用第 4 级。这比直接调模型摘要节省 90%+ 的压缩成本。

**3. 安全纵深防御**

```
用户命令 → Bash AST 解析 → 23 项静态检查 → 权限规则评估 → 用户确认 → 执行
                                                    ↓
                                              deny-first 拒绝
```

## 五、总结

这个项目的核心价值是**"知其然更知其所以然"**。

市面上有很多 AI 编程工具可以直接用，但：
- 直接用 Claude Code → 只会调 API，不理解架构
- 自己从零实现一个 → **掌握 Agent Loop、工具编排、安全防御、上下文管理的完整技术栈**

这些能力不会因为某个产品的更新而过时，因为它们是 AI Agent 的**底层设计模式**——就像理解了操作系统原理，用什么操作系统都能驾驭。
